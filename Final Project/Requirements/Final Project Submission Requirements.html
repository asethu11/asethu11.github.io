Final Project Submission

  • Due Dec 1 by 11:59pm
  • Points 200
  • Submitting a file upload
  • File Types zip
  • Available until Dec 3 at 11:59pm

 

This project gives you a chance to show off all that you've learned and it can be a lot of fun to see how far you've come since entering the program. Students who complete the course have enjoyed building this site and seeing how what you've learned about HTML and CSS can work with what you now know about JS. Have fun with it!

Your final project for this course requires you to use your knowledge of HTML, CSS and JS to build a single-page site that you will be able to include in your portfolio. Often when applying for jobs and internships you will be asked to share a completed project that you built and to speak about the work you did on that project. This assignment will be one that you can include in your portfolio. (You can view some single-page sites on the awwwards site).

The site can be for a real or fictional company, topic, or person. Depending on the type of job you hope to find after graduation, you may want to choose a subject that will illustrate the kind of work you can do. If you'd like to work in-house, building a site for a company could be the way to go. It can save time to choose an existing company, as they generally have established branding and content to pull from.

The Requirements

There will be three parts to this assignment:

  1. Your coded site, to include semantic HTML, appropriate CSS (your site is not required to be responsive), and JS. ALL of the code written for this page must be written from scratch by you (Do not copy code from the internet and include it in your project). Please see and review the academic integrity policy for more information.
    First, you'll have some required elements:
    • The site will be a single page, and should display properly at 1280px wide (fixed).
    • The page should look like a real, stand-alone website. Users should see all of the kind of content that they would expect from any professional website. No placeholder text is allowed, text and photos should be realistic and high-quality, and everything you'd expect to see should be present (header, navigation (even within the page), footer, etc).
    • The HTML code is semantic and uses appropriate HTML for any interactive elements (links, buttons, etc. are interactive elements, divs, spans, etc. are not).
    • Include a control and the proper functionality to allow the user to choose whether to display your site in light or dark mode. Some careful planning when choosing the colors and style for your site can make this very easy to implement. Watch a video demonstrating the light/dark mode functionality.
    • A contact form. This should have at least these inputs (coded as the correct types and labeled correctly):
      • full name (or first and last name separately) - marked required, validate with regex
      • phone number - only required if the users has a preference for being contacted by phone, validate with regex
      • email address - only required if the users has a preference for being contacted by email, validate with regex
      • comments - marked required
      • a radio set that allows the user to choose whether they'd like to be contacted by phone or email - required. (You can make one of these checked on page load).
      The name(s) and comments should be required, and the choice of how the user would like to be contacted should require either the phone or email (depending on what the user chose). The form should validate the required fields on submit (using JavaScript and regex, not HTML5 validation) and it should use the valid user info to create an object (customer/user/etc.) when the form is valid and submitted. Errors in the form should display to the user next to the input with the error and should be descriptive and noticeable. When the form is complete and submitted properly, the form should reset and display a message to the user thanking them for their submission, with the info they submitted, pulled from the object you created. This object/message can display on the page, in a modal, or in an alert, but the first two options are preferred.
  2. ALL of your files, including any images and other assets, JS, HTML or CSS should be well organized, properly indented, well-commented, and appropriate (in terms of content and layout). Follow the variable naming conventions I used in class to ensure that your JavaScript will still work after being obfuscated.
  3. Your HTML and CSS should validate without errors or warnings.
  4. There should not be ANY JavaScript or CSS in your HTML (no inline event handlers, no inline CSS, only HTML goes in your HTML file). If there is any JavaScript in your HTML a point deduction of 50% of the total possible score will be deducted from your final project grade. If that code is found to have been copied from elsewhere rather than being written from scratch by you, you will receive a zero on the project and an academic integrity violation referral. Again, this code should all have been written from scratch by you, with our in-class activities and zyBook acting as additional references and sources of code.
  5. When viewing your site in the browser, no errors should be visible in the console on load or after interacting with any of your JS controls.
  6. Add the async or defer attribute to your script tag in the header to prevent errors on page load with the JS. Your script tag linking to your JS file
  7. Your JS should "use strict"; globally
  8. Watch a video of general requirements on YouTube
  9. PLEASE DO NOT UPLOAD YOUR FILES TO GITHUB PAGES (OR ANY OTHER PUBLIC/VISIBLE LOCATION ONLINE UNLESS YOU OBFUSCATE YOUR JS (be sure to test after obfuscating to make sure your code still works. If you often forget things like semicolons at the ends of statements, or if you name your variables starting with uppercase letters, your code may not work after being obfuscated/minified).

These elements are required for the project/code as well, but you only have to choose to either include the product display and guessing game OR the cost calculator. The cost calculator is more difficult to implement, so you can choose to only add that feature to your site or you can choose instead to add the product display AND the guessing game:

    • Product Display - This would be a section of your page that allows the user to view some of the available products or services this company offers one at a time, by clicking on a control that includes the product name.
      • REQUIREMENTS:
        • Each product must have its own control with its name that is displayed at all times.
        • This is not a slideshow/carousel.
        • Each "product" must include a name, image, and short description (at least a sentence or two) that is displayed when chosen, and you should have a minimum of three products.
        • One product will display by default on page load, and only one product should ever display on screen at a time.
        • (Refer to Module 4 - Event Handling & The DOM for the code you will need to create this part of your project)
        • Watch a video demonstrating a product display
        • YOU MUST ALSO INCLUDE THE GAME PLAY ON YOUR SITE IF YOU CHOOSE THE PRODUCT DISPLAY FEATURE
    • Game Play - This section will most likely make sense as a sort of "contest" for the user.
      • There will be an input where the user enters a number between 1 and 10.
      • You will generate a random number that you display to the screen (with the one that the user guessed), and either declare them a winner (their guess matches the random number) or ask them to try again (their guess does not match the random number).
      • You don't have to actually do anything when they "win" except say that they won.
      • (Refer to Module 6 - Math Object, Module 4 - Event Handling & The DOM, Module 2 - Conditionals, possibly also M5 for validation for the code you will need to create this part of your project)
      • Watch a video demonstrating the game play
      • YOU MUST ALSO INCLUDE THE PRODUCT DISPLAY ON YOUR SITE IF YOU CHOOSE THE GAME PLAY FEATURE
    • Cost Calculator/Cart - This would be a section of your page that displays a collection of top products, probably for "quick order" or something similar.
      • Each product would include an image, price and name, and some control that the user can click on to add that product to their total, and you should include a minimum of three products.
      • The total/tax/shipping will display in the "total" area of the section, and the total cost should display the current product total, tax and/or shipping, and total price with tax on change (every time the control for a product is clicked, the total should update).
      • Include a button allowing the user to "check out" that when clicked will clear the cart, display the total cost of the items that were in the cart, and thank the user for their order.
      • If the cart is empty when the user clicks to checkout, they should see a message asking them to add something to the cart before they check out.
      • (Refer to Module 4 - Event Handling & The DOM, Module 1 - Arithmetic,  Module 3 - Arrays for the code you will need to create this part of your project)
      • Watch a video demonstrating the cost calculator/shopping cart
      • YOU DO NOT NEED TO COMPLETE THE GAME OR PRODUCT DISPLAY IF YOU CHOOSE THIS FEATURE AS IT IS MUCH MORE DIFFICULT
  2. A planned/prepared Behance project that explains your process. If you aren't sure what this should look like, check out the entries in the Digital Products category on Behance for GIT. This should include these details:
    1. Project requirements
    2. What the requirements inspired you to create
    3. An explanation of the goals/purpose/users of the site
    4. The steps you took when building the site (include any mockups or wireframes or brainstorming documents - wireframes OR mockups will be required, plan to complete at least one of these as part of your process) These should include: discovery/goals/content gathering, design, writing the code, testing the site design and code, final look/functionality. Include images and design elements in your Behance project!
    5. The tools you used, sources of images and other assets (include these in code comments too to be safe)
    6. Explanation of the next steps you would take and/or what you would do to add to or change the site with more time, what you struggled with, what you enjoyed the most, etc.
    7. PLEASE DO NOT INCLUDE SCREENSHOTS OF YOUR JS IN THIS PROJECT
  3. An actual presentation of your document/project as explained in item two above where you talk through the process and give a demo of your site. (This will happen during a class meeting. The order will be randomly generated and posted before the class meeting). This should be natural, where you use the prepared presentation/project like slides and talk about yourself and the site. You're the one that built it, now show off what you know!
    • You can find more information about talking about your process as a developer here (and a recording of this presentation on YouTube). As well as the a recording of the AIGA Poly Behance Workshop, and the slides used in that workshop. Some students are more comfortable creating their presentation in Google Slides and embedding that presentation in Behance, which is fine, but not ideal. The point is more to get you in the habit of talking about your process and adding things to your portfolio. The more detail and the better the design and code, the more you'll have to share with interviewers when you graduate!

Tips for Success

START EARLY! The sooner you start to think about this assignment, the easier it will be to build. Determining the site's subject/company is a big step, and so is gathering content. Take a few small steps every week, and decide how you can best show off your skills.

ASK FOR HELP IF YOU NEED IT! If you realize that you aren't sure how to implement some part of the site, just ask. I am here to help and I want to see everyone finish the course with an interactive site that shows off what you've learned in this class and in the rest of the program. If you're struggling or if you need some tips once you decide what to build, I will help. There are also multiple help/work days before this project is due that take place during our regular class time, so plan to attend them and ask questions.

DO SOME RESEARCH! This could be research into interviewing and sharing your portfolio as a designer/developer, or research into how to best design your presentation for the most impact, or even research into how to build a site with good SEO or how to build a site that meets your user's goals. This is a 400-level course, so you probably have built up some of this knowledge in your other courses, but it's always a good idea to ensure that you're up-to-date on the latest research related to building sites for the web.

The Submission:

  1.  Your single semantic HTML file, single CSS file (your site is not required to be responsive but the styles should be appropriate), and single (readable/non-obfuscated) JS file for your site, all coded from scratch by you, in a well organized project folder (compressed) along with all of the assets used on your site. This must be uploaded to Canvas correctly for credit. Do not link to a file hosted elsewhere. This code should work in exactly the same way/contain the same code as your uploaded site (except that the JS should not be obfuscated). 
  2. A link to your coded site, to include semantic HTML, appropriate CSS (your site is not required to be responsive), and (obfuscated) JS.
  3. A link to your Behance project.

This criterion is linked to a Learning Outcome JS - Element Choice 1 OR Cost Calculator _1308
The site includes one of the three features to choose from:
PRODUCT DISPLAY - A section on the page includes the option to view one of at least three items (products, etc.) at a time (separately from the cost calculator, if chosen). Each item should include an image, name, and description. Each product must have its own control with its name that is displayed at all times. This is not a slideshow/carousel. Only one product and its info should display at a time, and a control (button, etc.) is used by the user to choose which item to view, based on selecting that control. This is not a slideshow and should not autoplay. One product should display on page load (and only one) and it should be replaced by the product/control that the user clicks on (including the name, description and image).
COST CALCULATOR/CART - A section on the page that displays a collection of top products (separately from the product display, if chosen). Each product (minimum of three total) should include an image, price, name, and a control the user can click on in order to add that product to their cart/total. The user should also be able to remove the items from the cart/total and see an updated calculation each time this happens. The area that displays the total should also include the subtotal, tax, and shipping, and the total should reflect all of those values added together. The user should have the option to "check out" and when selected/clicked, the user should see a message showing what they ordered, their total, and thanking them for their order. If the cart is empty when the user clicks on that control, they should see a message asking them to add items to their cart before checking out.
GAME PLAY - A section on the page (separate from the others) where the user should see an input for a numerical value, as well as a button to click in order to play. The game should accept a number from the user between one and ten, and should generate a random number between the same values to compare it to. If the two values match, the user "wins" and if they don't, the user does not win. Both numbers (the user input and the random number) should be displayed to the user on click, as well as a message saying whether or not the user won the game and asking them to try again if they lost.
